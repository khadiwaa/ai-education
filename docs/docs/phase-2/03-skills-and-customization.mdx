# Module 3: Skills, Extensions & Customization

*Phase 2 · ~60–75 minutes · Practical*

This module covers how teams make Copilot feel less generic.

Out of the box, Copilot is good at broad software tasks. It becomes much more valuable when you add **team context**, **repo-local behavior**, and **custom tools** that reflect how your organization actually works.

For this curriculum, you do **not** need to learn how to write extension code to benefit from it. The important skill is understanding what customization mechanisms exist, what they are good for, and how to use the ones already present in your repo.

---

## What Copilot CLI skills and extensions are

In practical terms:

- a **skill** is a specialized behavior or workflow Copilot can invoke
- an **extension** is one way to package tools, hooks, and logic that give Copilot that behavior

From an engineer’s point of view, the key idea is simple: **your team can teach Copilot how to work better in your environment**.

### What a skill can do

A skill can:

- detect trigger phrases
- tell Copilot which tools to call
- ask you structured questions
- save a file
- call an API
- enforce a workflow shape
- produce output in a team-standard format

That means skills are useful for things like:

- onboarding flows
- assessment flows
- release checklists
- incident triage helpers
- documentation generators
- internal API lookup tools

### Skills vs. instructions vs. MCP

| Mechanism | Best for | Example |
|---|---|---|
| **Custom instructions** | Stable guidance Copilot should always know | naming conventions, preferred test framework |
| **Skills / extensions** | Repeatable workflows or custom tools | “assess me”, “generate release notes”, “run architecture checklist” |
| **MCP servers** | External tools and systems | GitHub, Slack, databases, internal APIs |

## The `.github/extensions/` directory

Repo-local extensions live in:

```text
.github/extensions/
```

In this repository, there is already a working example:

```text
.github/extensions/
  ai-assessment/
    README.md
    extension.mjs
```

### Why this matters

This is how a team can ship Copilot behavior **with the repository**.

When someone works in the repo, the customization travels with it. That means:

- the workflow is discoverable
- the behavior is versioned in Git
- improvements happen through normal pull requests
- everyone on the repo can benefit from the same capability

:::tip Team-level leverage
A good repo-local extension is a force multiplier. One person builds a useful workflow once; everyone else gets it “for free” every time they use Copilot in that repo.
:::

## Using the assessment skill in this repo

This repo includes an assessment skill for the **Phase 1 self-assessment**.

You can trigger it with prompts like:

```text
assess me
quiz me
run the AI knowledge check
test my understanding
```

You can also scope it:

```text
assess me on module 1
quiz me on using AI effectively
run the assessment for module 3
```

### What happens when you use it

The skill will:

1. ask for your name
2. run a structured assessment across Phase 1 concepts
3. score each answer
4. give brief feedback after each response
5. save a markdown report file at the end

The report is written to:

```text
assessments/<your-name>-ai-assessment-<YYYY-MM-DD>.md
```

That directory is ignored by Git in this repo, so the report is personal unless you choose to share it.

### What the report contains

- question-by-question scores
- strengths
- development areas
- recommended learning topics
- a recommended follow-up module

:::info Why this is a good teaching example
The assessment skill shows what a useful team extension looks like:

- it triggers from natural language
- it guides a conversation step by step
- it uses tools behind the scenes
- it produces a persistent output artifact
:::

## How keyword hooks work

One common pattern in skills is a **keyword hook**.

In plain English: when your prompt matches certain phrases, the extension adds hidden guidance telling Copilot which workflow to run.

In this repo’s assessment extension, the trigger phrases include patterns like:

- `assess me`
- `quiz me`
- `knowledge check`
- `test my understanding`
- `run the assessment`
- `ai assessment`

That means the engineer does **not** need to know the underlying tool names. They just talk naturally.

### Why keyword hooks are useful

They reduce ceremony.

Instead of saying:

```text
Invoke the assessment workflow, call the starter tool, ask me each question one at a time, then save a report.
```

you can say:

```text
assess me
```

### But be careful

Hooks should be:

- easy to discover
- narrow enough to avoid accidental triggers
- documented in the repo

If a hook is too broad, it becomes surprising and annoying.

## What skills can do without you writing code

You do not need to author extensions to benefit from them, but you should understand the capability surface.

### Skills can orchestrate tools

For example, a skill can tell Copilot:

- ask the user a few questions
- collect structured answers
- call a save-report tool
- write a markdown result into the repo

### Skills can shape the conversation

They can instruct Copilot to:

- ask one question at a time
- score answers against a rubric
- respond in a standard format
- avoid showing hidden evaluation guidance

### Skills can touch external systems

If the extension or connected tools allow it, a skill can:

- call a web API
- query GitHub
- fetch docs
- save artifacts
- kick off internal workflows

### What this means for engineers

A good skill can turn Copilot from “generic assistant” into “team-specific helper.”

Think:

- better onboarding
- repeatable assessments
- standard review templates
- guided release workflows
- less copy-pasting the same prompts every week

## Custom instructions: `.github/copilot-instructions.md`

Skills are not the only customization layer. For many teams, the highest-value customization is still the simplest one: a repo-level instructions file.

Path:

```text
.github/copilot-instructions.md
```

### What it is

This is a Markdown file containing repo-specific guidance that gets injected into Copilot interactions in supported tools, including VS Code and the newer Copilot CLI.

:::note `AGENTS.md` — the tool-agnostic cousin
You'll also see a root-level **`AGENTS.md`** in many repos (the [agents.md](https://agents.md) convention). It's the same idea as `.github/copilot-instructions.md` — a Markdown "README for AI agents" — but it's *tool-agnostic*: editors and agents beyond GitHub's (Cursor, Codex, Aider, Zed, and others) look for it. The practical move: pick **one** file as the source of truth and have the other point to it, so the two never drift. **This very repo does exactly that** — read [`.github/copilot-instructions.md`](https://github.com/alexkhadiwala-hl/ai-education/blob/main/.github/copilot-instructions.md) and the root [`AGENTS.md`](https://github.com/alexkhadiwala-hl/ai-education/blob/main/AGENTS.md) as a real example.
:::

It is the right place for information like:

- project purpose
- preferred libraries
- naming conventions
- testing expectations
- architectural boundaries
- build/validation commands

### What it is not

It is **not** a policy engine.

It does not magically enforce correctness. It improves the quality of the starting point by telling Copilot what “normal” looks like in this repo.

### Format

It is just Markdown. No JSON schema required.

Example:

```md
# Copilot Instructions

## Code shape
- Prefer small, reviewable patches.
- Reuse existing helpers before introducing new abstractions.
- Preserve public API shapes unless explicitly asked to make a breaking change.

## Testing
- Add or update tests when behavior changes.
- Use Jest for unit tests and keep test files co-located with the source they cover.

## Validation
- Run the relevant test suite before proposing completion.
- Summarize what was validated and what was not.
```

### Good use cases

- coding standards
- naming conventions
- preferred libraries
- project architecture notes
- “do not do X” guidance
- build/test/run instructions

### Poor use cases

- task-specific one-offs
- giant walls of text nobody will maintain
- conflicting instructions for different directories without path-specific files

### Team benefits

This is one of the best “small investment, broad payoff” improvements a team can make.

Benefits:

- everyone gets the same baseline guidance automatically
- prompts become shorter
- generated code becomes more consistent
- new engineers avoid common repo-specific mistakes
- review comments about known conventions happen less often

:::tip Keep it concise and operational
The best instruction files read like notes from a strong staff engineer:

- what this repo is
- where important things live
- how to validate changes
- what conventions matter most
:::

## Workspace configuration: helping Copilot get good context

Even without writing a single extension, you can improve Copilot’s output by how you organize the workspace and the repo.

### Keep the right files open

If you are editing a handler, it often helps to keep open:

- the handler you are changing
- a similar existing handler
- the shared validator or helper it should reuse
- a representative test file

That gives Copilot strong local pattern information.

### Use good naming conventions

Names are context.

- `retry_payment_capture` is better than `do_stuff`
- `UserQuotaExceededError` is better than `CustomError`
- `normalize_input_rows` is better than `helper`

### Write descriptive READMEs and comments

Copilot reads the repo the way engineers do: from the artifacts available.

High-value artifacts include:

- README sections that describe architecture and setup
- comments that explain invariants or surprising behavior
- examples of the preferred pattern in tests or modules
- docs for commands that actually work

### Organize by stable patterns

Repositories that are easier for humans to navigate are also easier for Copilot to help with.

Signals that help:

- predictable folder names
- obvious entry points
- canonical examples
- clear module boundaries

Signals that hurt:

- inconsistent naming
- duplicate patterns with different conventions
- stale READMEs
- hidden build steps known only by tribal memory

## Finding community skills and extensions

You do not need to invent everything yourself.

Good places to look:

- GitHub repos that demonstrate Copilot CLI extensions
- GitHub Docs examples for skills, custom instructions, and agents
- your internal developer platform or enablement teams
- communities sharing MCP servers and editor customizations

### How to evaluate a community extension

Before installing or depending on one, ask:

- What exact problem does it solve?
- Is it maintained?
- What permissions does it need?
- Does it read or write files?
- Does it call external services?
- Is the behavior documented clearly enough that teammates can trust it?

### A good adoption rule

Do not add a skill because it is clever. Add it because it removes repeated friction for many engineers.

Examples of genuinely good candidates:

- assessment or onboarding flows
- release-note generation
- repo-specific bootstrap help
- architecture summary or discovery helpers
- internal service lookup or schema lookup

## A simple customization strategy for teams

If your team is just getting started, the order of operations should usually be:

1. Add `.github/copilot-instructions.md`
2. Improve README/setup docs and canonical examples
3. Add one high-value repo-local extension for a repeated workflow
4. Add MCP integrations only when you need external systems

This order works because:

- instructions improve every interaction
- better docs improve both human and AI understanding
- one good extension proves the pattern
- MCP is powerful but introduces security and governance concerns

## Key Takeaways

1. **Skills and extensions make Copilot repo-specific.** They are how teams package workflows, tools, and triggers into something reusable.
2. **This repo already includes a real example.** The AI assessment skill shows how natural-language triggers can launch a structured workflow and save a report.
3. **Custom instructions are the simplest high-leverage customization.** `.github/copilot-instructions.md` improves every Copilot interaction in the repo.
4. **Workspace quality is context quality.** Good names, READMEs, patterns, and open files all improve output.
5. **Adopt customizations for repeated friction, not novelty.** The best ones save time for many engineers repeatedly.
