# Module 2: Copilot in VS Code

*Phase 2 · ~75–90 minutes · Practical*

This module is about using Copilot well **inside the editor**. The editor is where Copilot becomes most powerful for day-to-day engineering work, because it can see your current file, nearby files, selections, diagnostics, and terminal output.

The key idea is simple:

- **Inline completions** are for finishing what you are already writing.
- **Copilot Chat** is for questions, debugging, and scoped generation.
- **Copilot Edits** is for reviewable multi-file changes.

---

## The Copilot extension ecosystem in VS Code

Most engineers will encounter two extensions:

| Extension | What it does | When you feel it |
|---|---|---|
| **GitHub Copilot** | Inline code completions and suggestions in the editor | While typing |
| **GitHub Copilot Chat** | Chat panel, slash commands, code-aware conversations, Edits/Agent workflows | When asking questions or requesting changes |

Think of them as complementary, not competing.

- **Copilot** helps you continue the line, block, or function you are already shaping.
- **Copilot Chat** helps you step back, discuss intent, inspect code, or ask for generated changes.

:::tip Mental model
Inline completion is like a very fast pair programmer finishing your sentence.

Chat is like asking that pair programmer a question.

Edits is like asking them to propose a patch and then handing you a diff for review.
:::

### Same model family, different context surface

Engineers sometimes ask why Copilot “feels smarter” in the editor than in the terminal.

Usually the answer is not the model. The answer is context.

In VS Code, Copilot can often see:

- the file you are editing
- nearby symbols and imports
- other open tabs
- selected code
- workspace search results
- diagnostics and terminal output in chat workflows

That is why editor prompts often need less pasted context than terminal prompts.

## Inline completions

Inline completions are the gray suggestions that appear directly in your editor as you type.

### How inline completions trigger

They usually appear when Copilot has enough local signal, for example:

- a clear function name
- a useful comment above a function
- a partially written control-flow block
- existing patterns elsewhere in the file
- imports, types, and variable names that make the intent obvious

Inline completion is strongest when **you have already done the thinking** and left a strong trail in the code.

### Important inline completion actions

| Action | Common shortcut | Notes |
|---|---|---|
| Accept suggestion | <kbd>Tab</kbd> | The most important one |
| Dismiss suggestion | <kbd>Esc</kbd> | Useful when a bad suggestion keeps reappearing |
| Next alternative | <kbd>Alt</kbd>+<kbd>]</kbd> / <kbd>Option</kbd>+<kbd>]</kbd> | Cycle forward |
| Previous alternative | <kbd>Alt</kbd>+<kbd>[</kbd> / <kbd>Option</kbd>+<kbd>[</kbd> | Cycle backward |
| Partial accept | <kbd>Ctrl</kbd>+<kbd>Right</kbd> or platform-specific equivalent | Accept the next chunk/word instead of the whole suggestion |

:::note About keybindings
Exact defaults can vary by operating system, keyboard layout, keymap extension, and local customizations. If one of these is not working, open **Keyboard Shortcuts** in VS Code and search for “Copilot”.
:::

### Why partial accept matters

Partial accept is underrated.

Often Copilot gets the first 60% right and the last 40% wrong. If you always accept or reject whole suggestions, you throw away useful work. Partial accept lets you keep the good prefix and keep typing.

### Best practices for inline completions

#### Start with comments, signatures, and names

Poor setup:

```typescript
function handle(data: any) {
    
}
```

Better setup:

```typescript
/**
 * Returns unique, lowercased emails in original order. Ignores null and empty values.
 */
function normalizeCustomerEmails(records: CustomerRecord[]): string[] {
```

The second version gives Copilot:

- intent
- shape of data
- expected return type
- important edge cases

#### Write the first line or two yourself

If you sketch the structure, Copilot usually snaps to the pattern.

```ts
export function formatCurrency(amountCents: number, currency: string): string {
  const amount = amountCents / 100;
}
```

That first line is often enough for Copilot to fill in an `Intl.NumberFormat` implementation.

#### Use descriptive names

`process()`, `handle()`, and `doThing()` are weak signals.

`reconcileInvoiceStatus`, `buildRetryBackoffSchedule`, and `parseReleaseMetadata` are strong signals.

#### Keep relevant files open

If Copilot needs to mirror an existing pattern, open the source file that already contains the pattern. You are effectively curating context.

#### Accept less, shape more

The fastest engineers do **not** accept giant suggestions blindly. They:

- type the important parts
- let Copilot fill in the boilerplate
- reject when it starts drifting
- keep the local code shape under human control

## When inline completions are the wrong tool

Do **not** force inline completion to solve tasks that are really questions.

Use Chat instead when you need to ask things like:

- “Where in this codebase is auth handled?”
- “Why is this failing?”
- “What tests should exist for this?”
- “Refactor this across several files.”

A quick rule:

- **Inline** → continue writing
- **Chat** → ask or discuss
- **Edits** → propose coordinated code changes

## Copilot Chat panel

Copilot Chat is where you switch from “finish my code” to “help me think.”

Typical use cases:

- ask about a file, symbol, or failure
- request tests or docs for selected code
- compare two approaches before editing
- generate a draft implementation plan
- summarize terminal output
- scope a refactor before you touch multiple files

### Chat vs. inline: a practical distinction

| Use case | Better tool |
|---|---|
| Finish a loop, function body, or small block | Inline completion |
| Explain code you are reading | Chat |
| Generate tests for a selected function | Chat |
| Ask a codebase-wide question | Chat with `@workspace` |
| Make a reviewable multi-file patch | Edits |

### Good chat prompts

```text
Explain the error-handling strategy in #selection.
```

```text
@workspace Where is rate limiting enforced for outbound API calls?
```

```text
@file Compare this file to the service pattern used elsewhere in the repo. What is inconsistent?
```

```text
#terminal What is the first actionable error here, and what should I inspect next?
```

## Context variables in Chat

Context variables are one of the biggest productivity unlocks in VS Code Chat because they let you ground the prompt without copy-pasting everything.

### `@workspace`

Use when the question is about the **whole codebase**.

Good examples:

```text
@workspace Where is request validation performed for user creation?
```

```text
@workspace Summarize how background jobs are scheduled and retried.
```

Best for:

- architecture questions
- “where is X handled?”
- tracing flows across files
- finding existing patterns before adding new code

### `@file`

Use when one specific file matters most.

```text
@file Explain the invariants in this module.
```

```text
@file Suggest the smallest safe refactor to make this easier to test.
```

Best for:

- file-level explanation
- local refactors
- asking for tests or docs for one module

### `#selection`

Use when you want precision.

```text
/fix #selection Make this logic handle empty arrays and None values safely.
```

Best for:

- explanation of a specific function
- tests for one block
- refactoring a single region
- docstring generation for a selected symbol

### `#terminal`

Use when the terminal output matters.

```text
#terminal Explain this test failure and tell me the most likely root cause.
```

```text
#terminal I ran the migration and got this SQL error. What should I inspect first?
```

Best for:

- test failures
- build failures
- compiler errors
- runtime stack traces

### `@github`

Use when GitHub artifacts are part of the task.

```text
@github Summarize the linked issue and suggest an implementation plan.
```

```text
@github What changed in the PR discussion that affects this file?
```

Best for:

- issue/PR context
- reviewer discussion
- connecting code changes to ticket intent

### Quick context-variable table

| Variable | Scope | Best for |
|---|---|---|
| `@workspace` | Whole repo/workspace | Architecture, discovery, tracing |
| `@file` | Specific file | Explanations and scoped questions |
| `#selection` | Selected code | Precise fixes, docs, tests |
| `#terminal` | Terminal output | Debugging build/test/runtime failures |
| `@github` | Issues, PRs, GitHub artifacts | Linking code to discussion and ticket context |

## Slash commands in Chat

Slash commands give you structured shortcuts for common requests.

| Command | What it is good for | Example |
|---|---|---|
| `/explain` | Explain code in plain English | `/explain #selection` |
| `/fix` | Suggest fixes for a problem area | `/fix #selection handle nil input safely` |
| `/tests` | Generate tests | `/tests #selection use Jest` |
| `/docs` | Generate or improve docs/comments | `/docs #selection` |
| `/new` | Create a new file from an instruction | `/new Create a migration script for archiving stale sessions` |
| `/newNotebook` | Create a notebook | `/newNotebook Explore these CSV metrics` |

:::tip A simple heuristic
Start with a slash command when you know the task category.

- explanation → `/explain`
- repair → `/fix`
- tests → `/tests`
- documentation → `/docs`

Then add normal natural-language constraints after it.
:::

### Better slash-command prompts

Weak:

```text
/tests
```

Better:

```text
/tests #selection Use Jest. Include empty input, duplicate values, and invalid timestamp cases.
```

Weak:

```text
/fix
```

Better:

```text
/fix #selection Preserve the public API. Do not add dependencies. Prefer the smallest safe patch.
```

## Copilot Edits

Copilot Edits is the right tool when the task spans multiple changes and you want to **review a diff before accepting it**.

Think of Edits as the bridge between “chat about it” and “actually propose the patch.”

### When to use Edits

Good fits:

- rename a pattern across a few files
- update imports after moving code
- add tests plus the production change together
- apply a repeated API migration in a controlled scope
- implement a moderate feature with reviewable changes

Bad fits:

- tiny one-line completions
- repo-wide refactors with unclear requirements
- risky architectural rewrites without a plan

### A strong Edits workflow

1. **Scope the change tightly.** Say what should change and what should not.
2. **Reference the right files.** Open or cite the files that matter.
3. **Ask for the smallest coherent patch.** Avoid “rewrite everything” prompts.
4. **Review the diff carefully.** Especially consistency across files.
5. **Accept or reject intentionally.** Do not accept a big patch because the first 80% looks fine.

### Example Edits prompt

```text
Update the user creation flow to validate email format before persistence.
Scope:
- change only the handler, validator, and tests
- preserve the response shape
- use the existing validation helper if possible
- add tests for invalid and missing email
```

### What to review in the diff

- Did it change files outside the intended scope?
- Did it preserve naming and local conventions?
- Did tests actually cover the changed behavior?
- Did it leave TODOs or partial updates behind?
- Did it create inconsistent patterns across files?

:::warning Multi-file changes deserve skepticism
Copilot can be locally correct and globally inconsistent. A patch that looks fine in each file can still drift across naming, abstractions, or error handling conventions.
:::

## Inline vs. Chat vs. Edits

This is the decision most engineers need to get right.

| If you need to… | Use |
|---|---|
| Finish code you are already writing | Inline completion |
| Ask a question about code | Chat |
| Debug from logs or terminal output | Chat with `#terminal` |
| Search the repo conceptually | Chat with `@workspace` |
| Make a reviewable multi-file patch | Edits |
| Do a giant refactor | Break it into steps; Edits or CLI agent per step |

### The wrong-tool smell

If you are typing a paragraph into the editor and waiting for inline completion to somehow infer the whole task, switch to Chat.

If you are asking Chat to mutate five files but never reviewing a diff, switch to Edits.

Below are the most useful shortcuts to internalize first.

| Action | Common shortcut | Why it matters |
|---|---|---|
| Accept inline completion | <kbd>Tab</kbd> | Fastest path for boilerplate and continuation |
| Dismiss inline completion | <kbd>Esc</kbd> | Stops bad suggestions from hijacking flow |
| Next inline alternative | <kbd>Alt</kbd>+<kbd>]</kbd> / <kbd>Option</kbd>+<kbd>]</kbd> | Explore alternatives without rewriting prompt |

### Suggested personal shortcut hygiene

If you use Copilot heavily, consider giving yourself easy keybindings for:

- opening Copilot Chat
- starting Copilot Edits
- toggling inline suggestions
- accepting next word/partial suggestion

Search **Keyboard Shortcuts** for “Copilot” and customize the commands you use most.

## Copilot settings that matter

There are two levels of settings to care about:

- **feature on/off settings**
- **context quality settings**

### Enable or disable by language

One of the most useful controls is limiting where Copilot is active.

Example `settings.json` pattern:

```json
{
  "github.copilot.enable": {
    "*": true,
    "plaintext": false,
    "markdown": true,
    "scminput": false
  }
}
```

Why do this?

- turn it off in low-value surfaces like commit message inputs if you do not want noise
- keep it on in code and documentation where it is actually useful

### Enable suggestions for comments and intent-signaling text

Many of Copilot’s best completions start from comments or docstrings. In settings, look for Copilot options related to editor suggestions and comment-driven completions, and leave them enabled unless they are creating noise.

### Telemetry and policy awareness

Depending on your organization, some telemetry and Copilot settings may be centrally managed. At the local level:

- review the Copilot extension settings
- review VS Code telemetry/privacy settings
- know whether your org has custom instructions, policies, or model restrictions enabled

:::note Team environments vary
If a setting described in public docs is missing or locked, your organization may be managing it centrally. In that case, document the team standard instead of fighting local overrides.
:::

## Tips for getting better completions

These matter more than prompt cleverness.

### Open the files that matter

If you are implementing a new endpoint and there is already a similar one elsewhere, open both files. You are effectively telling Copilot: “match this pattern.”

### Write descriptive names first

Names are cheap context and high leverage.

```typescript
function retryWithExponentialBackoff(...)
```

is better than:

```typescript
function helper(...)
```

### Use comments to specify behavior, not implementation trivia

Good:

```typescript
// Return cached data when still fresh; otherwise fetch and refresh the cache.
```

Weak:

```typescript
// do cache stuff
```

### Keep examples nearby

If your repo has a canonical test pattern, open one of those tests before asking for `/tests`.

### Stop when drift starts

If Copilot begins inventing APIs or changing patterns you did not ask for, stop accepting suggestions. Re-anchor the task with clearer context or switch tools.

## `.github/copilot-instructions.md` for workspace-level customization

This file is one of the highest-leverage team tools you can add.

It lives at:

```text
.github/copilot-instructions.md
```

Copilot reads it as repo-level guidance. It is the right place for stable instructions you do **not** want every engineer to repeat in every prompt.

### Good things to put in it

- naming conventions
- preferred frameworks and libraries
- testing expectations
- “smallest safe patch first” guidance
- repo-specific architecture notes
- commands for build/test/lint

### Example

```md
# Copilot Instructions

- Prefer small, reviewable patches.
- Do not add new dependencies unless explicitly asked.
- Use Jest for unit tests; keep test files next to the source files they cover.
- Reuse existing validation helpers before creating new ones.
- Preserve API response shapes unless the prompt explicitly asks for a breaking change.
```

### Why teams should care

Without custom instructions, every engineer has to remember to say the same things over and over.

With them:

- prompts get shorter
- suggestions get more consistent
- new engineers ramp faster
- code review noise drops because Copilot starts from the team standard

## A practical editor workflow

Here is a realistic pattern that works well.

1. Open the relevant implementation file, a similar existing file, and a representative test file.
2. Write the function name, signature, and one-line behavior comment.
3. Use inline completion for the obvious scaffolding.
4. Switch to Chat for questions or edge cases.
5. Use `/tests` or Copilot Edits for reviewable generated tests.
6. Review all generated code before you move on.

### Example sequence

```text
1. Write a strong function signature.
2. Accept a small inline suggestion.
3. Ask Chat: /tests #selection use Jest and include empty input + duplicate values.
4. Ask Chat: /docs #selection.
5. Review the diff, run tests, then edit manually.
```

---

## Key Takeaways

1. **Inline completions, Chat, and Edits are different tools.** Use the right one for the shape of the task.
2. **The model completes from context.** Strong names, comments, selections, and open files matter more than clever phrasing.
3. **Context variables are a force multiplier.** `@workspace`, `@file`, `#selection`, `#terminal`, and `@github` reduce copy-paste and improve grounding.
4. **Edits is for reviewable multi-file change.** Treat the diff as a proposal, not an answer.
5. **Repo-level instructions improve consistency.** `.github/copilot-instructions.md` is one of the best team investments you can make.
